Webex Connect Lab

Instructor Guide — Farmers Insurance Renewal Outreach

Estimated delivery time: 60 minutes | Audience: same as Lab 1 (technical & business leads)

This is Lab 2 in a 2-lab series. Lab 1 (Webex AI Agent Studio) builds the conversational brain. Lab 2 (this one) builds the orchestration that delivers proactive SMS, branches on customer reply, and hands off to the AI Agent.

The strongest narrative is to deliver them back-to-back: Lab 1 in the morning, Lab 2 in the afternoon. Together they tell the full "build → deploy" story.

One unverified assumption underpins this lab: the Webex Connect AI Agent node supports SMS as the channel (not just Live Chat). Public documentation as of design time states the AI Agent node is web-chat-only. The lab design relies on you having confirmed in your tenant that the SMS channel is supported. Do this validation in the dry-run before you deliver.

If the AI Agent SMS node turns out to be unavailable, the participant guide includes a documented fallback: replace the AI Agent branch with a cosmetic SMS like "A virtual agent will reach out shortly." The flow logic is identical; only the demo magic is reduced.

Tenant & Account Setup

The lab uses a dedicated Webex Connect tenant provisioned for the Farmers account team. All 10 participants log into the same tenant with seat-numbered accounts (labuser01–labuser10) and build their own flows side-by-side.

Per-Seat Accounts

Provision 10 user accounts in the tenant. The username/password values must match what's in index.html's labUsers array. Update both in lockstep.

  • 10 user accounts created in the tenant
  • All accounts assigned to the same service / workspace where flows will live
  • All accounts have permission to create + publish flows, view logs, and use the SMS sender
  • Credentials match the labUsers array in index.html

If your tenant doesn't support 10 separate users, you can fall back to 1 shared login + 10 named flows (e.g. Renewal Flow 1, Renewal Flow 2...). The tradeoff: participants can see and accidentally edit each other's flows. Watch for this during delivery.

SMS Sender & Country Support

Confirm before the lab:

  • The tenant has a long code or short code provisioned that all participants can select as "From Number"
  • The sender supports SMS to the country code(s) all participants will use (typically US/Canada)
  • The sender supports two-way SMS (inbound replies) — not just outbound
  • You've sent a test SMS to your own phone and received it within 5 seconds
  • You've replied to the test SMS and confirmed the inbound was received in the flow's execution log

If any participant has a non-US phone number, verify their country is supported by the sender BEFORE the lab starts. Discovering at minute 50 that 3 participants can't receive SMS will derail the demo. Prepare a fallback (e.g. they pair with a participant whose number works, or use a pool number).

Static Trigger Page

The static "renewal-trigger" web page is the participant's manual trigger for the flow. They paste their webhook URL + their phone number into a form, click a button, and the page POSTs the renewal event to their Connect flow.

The page is bundled in the lab guide repo and deployed to the same Render static site. URL: https://farmers-webex-connect-lab-guide.onrender.com/renewal-trigger.html (or simply renewal-trigger.html as a relative link from any lab guide page).

  • Static page is deployed and reachable
  • Page form fields: webhook URL, mobile number, (optional) Service Key header value
  • Page POSTs JSON: { "phone_number": "...", "policy_id": "FRM-101", "vehicle": "2022 Toyota RAV4" }
  • Page displays the HTTP status code and response body so participants can self-debug 401 / 404 errors
  • You've tested the page yourself end-to-end

Pre-Built AI Agent

The "virtual agent" branch in the lab hands off to a pre-built Webex AI Agent. You must build this agent before the lab, name it consistently, and confirm participants can reference it in their flow's AI Agent node.

Recommended Agent Configuration

  • Agent name: farmers-renewal-agent (or another name documented in the participant guide)
  • Agent persona: friendly Farmers renewal specialist
  • Agent knowledge: renewal eligibility, coverage options, payment plans (lightweight; doesn't need to be exhaustive)
  • Agent published and reachable via the AI Agent node in Connect
  • If using SMS channel: confirmed working with at least one test SMS conversation

The participant guide treats this agent as given. Participants don't build or modify it. Their job is just to wire the AI Agent node in their flow to point at this agent.

Dry-Run Validation

Run the entire lab end-to-end yourself, in the actual lab tenant, before delivery. The participant guide was drafted from public documentation, not hands-on tenant testing — there will be small UI label mismatches and possibly larger functional gaps.

Critical Validations

  1. Inbound Webhook node: confirm the exact menu path, the exact node name in the palette, and the URL format the node generates after save
  2. Variable syntax: confirm whether participants reference the inbound payload as $(node.trigger.phone_number), $(n1.inbound.phone_number), or some other syntax in this tenant version
  3. SMS node: confirm the From Number dropdown shows the lab's pre-provisioned sender
  4. Receive node: confirm the timeout field accepts seconds (300) and the captured-message variable name
  5. Branch node: confirm "Equals (ignore case)" is the actual operator label in the tenant UI
  6. AI Agent node — CRITICAL: confirm the node accepts SMS as the channel and successfully hands off to the pre-built agent. This is the assumption most likely to break.
  7. Delay node: confirm 30-second delay works and doesn't throw a "minimum delay" error
  8. End-to-end test: trigger from the static page, receive SMS, reply each option (YES, 1, 2), confirm each branch executes correctly

If a UI label or behavior differs from what the participant guide says, update the guide before delivery — don't ask participants to mentally map the docs to what they see. The Lab 1 dry-run revealed half a dozen UI drift issues, and patching them in the guide saved significant in-lab confusion.

Capture Screenshots During Dry-Run

For each of the 7 modules, take an annotated screenshot of the most likely point of confusion. Drop them in ~/Downloads/screenshots/, name them descriptively (e.g. module2-step3-webhook-properties-panel.png), and ask the orchestrator to embed them in the participant guide. Run through this 30 minutes before the lab starts:

  • Webex Connect tenant is reachable; you can log in
  • All 10 lab user accounts work (test 1, trust the rest)
  • SMS sender works: outbound + inbound
  • Pre-built AI Agent is published and reachable from a test flow
  • Static trigger page is deployed and POSTs successfully to a test flow's webhook
  • Dry-run executed end-to-end at least once today
  • Lab guide URL works: https://farmers-webex-connect-lab-guide.onrender.com

  • You've verified each participant's mobile number country is supported by the SMS sender

Module 2: Create the Flow + Webhook Trigger 10 min

Cumulative: 0:10 → 0:20

Each participant creates an empty flow named Renewal Flow N and configures the Inbound Webhook trigger node.

Expected Sticking Points

  • "Where do I create a flow?" — confirm the menu path during dry-run; many tenants label this differently (Service Designer / Flow Designer / Flow Builder)
  • "Which node is the trigger?" — point them to "Inbound Webhook"; some palettes group it under "Triggers" or "Start"
  • "Where's my webhook URL?" — they need to save the flow first; the URL only appears after the first save

Talking Points

"Every flow you build gets its own unique webhook URL — that's how you isolate your work from the other participants without needing separate tenants. Your URL is yours alone."

Module 3: Send the Renewal SMS 10 min

Cumulative: 0:20 → 0:30

Add an SMS node, configure From + To + Message, wire it to the trigger.

Expected Sticking Points

  • Variable picker: participants need to insert $(node.trigger.phone_number) into the To field. The flow builder has a variable picker — show it; don't ask them to type the syntax from memory
  • From Number dropdown: confirm the dropdown shows the lab sender; if multiple senders are listed, tell them which one to pick
  • Message body: provide a copy/paste-able starter message in the participant guide

Talking Points

"This is the moment Connect earns its keep. With one node, you've sent a real SMS to a real phone — no SMS gateway integration, no Twilio account, no carrier paperwork. The platform abstracts all of that."

Module 4: Wait for Reply + Branch on Response 15 min

Cumulative: 0:30 → 0:45

Add a Receive node (5-min timeout) and a Branch node with 3 conditions + default.

This Is the Hardest Module

Participants will struggle most here. Three distinct concepts collide: waiting for an async event, capturing its value, and conditional branching. Plan to spend the full 15 minutes and offer pair-help liberally.

Expected Sticking Points

  • Receive node placement: must come AFTER the SMS node, BEFORE the Branch — common mistake is to place it before SMS
  • Variable name for the captured reply: usually $(nX.receive.message) — the variable picker is your friend
  • Branch operator: "Equals" is case-sensitive; "Equals (ignore case)" is what they want for "YES"
  • Forgetting the Default branch: if a reply doesn't match any condition and there's no Default, the flow silently ends. Insist on a Default branch.

Talking Points

"The Receive node is what makes Connect a 'flow' platform instead of a 'send and forget' platform. It pauses your flow, possibly for hours, until the customer responds — without you managing any state."

Module 5: Configure Per-Branch Actions 10 min

Cumulative: 0:45 → 0:55

Wire each of the 3 branches to its outcome: confirm SMS, AI Agent handoff, human-agent SMS escalation.

Branch-by-Branch Detail

BranchNode SequenceNotes
YESSMS node (confirmation) → EndEasy. "Your renewal is confirmed."
1AI Agent node → EndConfigure the AI Agent node to point at farmers-renewal-agent.
2SMS (ask channel pref) → Receive → Branch (A/B) → SMS (confirm) → EndMini-flow inside the branch. Can simplify if running short on time.
DefaultSMS ("didn't understand") → EndCatch-all.

Branch 1 depends on the AI Agent node working with SMS. If your dry-run found that it doesn't, replace this branch with a cosmetic SMS in the participant guide and tell the room: "In production this would be a real AI Agent handoff — we're using a placeholder for the lab to keep the focus on Connect orchestration."

Time-Saving Move

If you're behind schedule, the easiest cuts are: (a) drop the channel-preference sub-flow in branch 2 (just send a single confirmation SMS), or (b) skip the Default branch (accept the small risk of silent failure).

Module 6: End-to-End Test & Wrap 5 min

Cumulative: 0:55 → 1:00

Each participant tests their flow with their own phone. This is the moment they'll remember.

Sequence

  1. Each participant copies their webhook URL from the trigger node
  2. Opens the static renewal-trigger page
  3. Pastes URL + their mobile number → clicks Send
  4. SMS arrives within ~5 seconds
  5. Replies YES (or 1 or 2) and watches the next SMS arrive
  6. Optional: try multiple replies in sequence to test all branches

Wrap Talking Points

"What you just built — proactive, two-way, branched, with AI handoff — would have taken weeks of integration work without a platform like Connect. We did it in 60 minutes."
"And every concept here scales horizontally — swap SMS for WhatsApp by changing one node. Add channel-preference logic. Add a delay so renewal reminders fire 30 days before expiry automatically. The flow is a starting point, not a destination."

Troubleshooting (Live Lab)

Participant clicks "Send" on static page → 401 Unauthorized
Service Key header value mismatch. Check the static page's Service Key field matches what the Inbound Webhook node expects. If the node was set to "No auth," tell them to remove the header entirely.
Participant clicks "Send" on static page → 404 Not Found
Webhook URL is wrong, or the flow isn't published. Have them re-copy the URL from the trigger node and verify the flow shows "Published" status.
Participant got the first SMS but their reply did nothing
Most likely the Receive node timed out before they replied (the default may have been 60 seconds; we recommend 300). Check the flow execution log to see exactly where it stopped. Also verify the Branch operator handles their reply's case (they replied "yes" but the branch is "Equals YES" case-sensitive).
SMS arrives garbled or with wrong content
Variable substitution failed. Look for the literal string $(node.trigger.phone_number) in the SMS body — that means the variable wasn't resolved. Have them re-insert via the variable picker.
Multiple SMSes arrive when only one should
Two paths in the flow are wired to SMS nodes that both fire. Have them check the flow diagram — likely a branch has 2 outgoing connectors instead of 1.
AI Agent node throws an error or doesn't hand off
Confirm the agent name in the node config exactly matches the published agent name. Confirm the agent supports SMS channel (this is the assumption flagged at the top of this guide). If unrecoverable, use the cosmetic-SMS fallback documented in the participant guide.

Common Questions From Participants

"Could we trigger this from a real event, not a web page?"

Yes — anything that can POST JSON can trigger the flow. In production, a renewal-management system fires the webhook 30 days before expiry. The static page in the lab is just a manual trigger for demo purposes.

"What if the customer doesn't reply for hours?"

Set the Receive timeout long (hours, even days). The flow waits without consuming compute. After timeout, the onTimeout branch fires — usually a fallback channel (email) or a callback to the renewal-management system.

"Can we A/B test the message?"

Yes — Connect has experiment/split nodes. Out of scope for this lab but a natural follow-on.

"How does this compare to building the same thing in Twilio Studio / Vonage / etc.?"

Connect's differentiators: native AI Agent handoff (single platform), tight Webex Contact Center integration (handoff to a real agent without leaving the platform), enterprise governance, and the unified Webex billing relationship. Twilio is more developer-flexible; Connect is more business-user-friendly and integrated.

"Why didn't we use the Webex Connect sandbox?"

The sandbox doesn't include the AI Agent node and lacks Live Chat. The lab tenant has the full feature set. If a participant wants to experiment after the lab, the sandbox is fine for SMS-only flows.

Lab Reset Procedure

Run this between sessions to reset the tenant to a clean state:

  1. For each Renewal Flow N: delete or unpublish
  2. Clear flow execution logs (optional; helps the next session debug their own runs without noise)
  3. Verify the pre-built AI Agent is still published (sometimes deletion of associated flows can affect it; usually safe but verify)
  4. Verify the SMS sender is still active and not over quota
  5. Reset any per-seat counters / flags if you've added them

Total reset time: ~5 minutes if no flows are deeply customized.
